Bookings

The main goal of GoneBusy is to connect users with each other via bookings. A Booking represents the commitment by one user to attend a Service of another user at a particular time. The API allows the creation and management of Bookings.

You can find all supported operations on the Bookings API page.

Categories

Categories are a way of organizing or tagging available Services.

You can find all supported operations on the Categories API page.

Resources

Resources are what actually provide Services. By default, every User is a User Resource for their own Services. You can create additional types of Resources for Services via the API.

Types of Resources

The following are the currently supported types of Resouces. If you have an idea for a Resource that can not be accommodated by any of the types below, please Contact Us.

  • Staff — create Staff Resources for each of the employees in your business, such as trainers at a gym, or therapists at a spa.

    first_name person's first name
    last_name person's last name
    gender person's gender

  • Place — a Place Resource would apply to something like a classroom, which hosts a particular Service at a time, or maybe a meeting room online where tutoring is offered.

    name the name of this Place
    capacity an optional limit to how many Bookings are allowed at a single time for this Place

  • Thing — a Thing Resource would apply to something like a van or equipment which is offered for rent for a certain amount of time.

    name the name of this Thing

You can find all supported operations on the Resources API page.

Services

Bookings on GoneBusy would not be possible without users offering Services. Services can be anything that a Resource performs, or provides, at a specific time for a particular duration.

Services can have a fixed duration when offered for booking, such as "1 Hour of Tutoring", or they can be booked in increments, such as a per-hour "Dogwalking" service. Whether a Service has a fixed duration or can be booked in increments of time is determined by it's PricingModel.

Follow this link for an example on how to create a Service.

You can find all supported operations on the Services API page.

PricingModels

Each Service must have a PricingModel associated with it in order to specify how to charge for the Service. There are several types of PricingModels that can be assigned to a Service.

When a Service is created without specifying a PricingModel, by default it will have an Activity PricingModel associated with it. To create a new PricingModel, one must specify the type of PricingModel (see below) along with any additional attributes the specific PricingModel requires.

If you need to price a Service in a manner that is not accommodated by any of the PricingModels below, please let us know and we’ll be happy to assist you in creating a new PricingModel if necessary.

You can find all supported operations on the Pricing Models API page.

Types of PricingModels

  • Activity — when a Service is free of charge, such as an activity you only share with friends or work colleagues, you can specify the Activity PricingModel. Note that this is the default PricingModel associated with a new Service upon creation.

    name (optional) an optional name for the PricingModel
    notes (optional) an optional notes field for the PricingModel

  • ByTheHour — when Services are charged for each hour being reserved, specify the ByTheHour type when creating a PricingModel for the Service. There are many examples of businesses that offer Services or classes by the hour, such as salons, yoga studios, guitar lessons, etc.

    price price that will be charged for each hour reserved
    currency (optional) currency of price. Defaults to USD
    name (optional) an optional name for the PricingModel
    notes (optional) an optional notes field for the PricingModel

  • ByTheMinute — when Services are charged for each minute being reserved, specify the ByTheMinute type when creating a PricingModel for the Service. Examples of Services that may charge by the minute may include tutoring, a help line, etc. Note that currently, the minimum allowable time to book a Service on GoneBusy is 15 minutes.

    price price that will be charged for each minute reserved
    currency (optional) currency of price. Defaults to USD
    name (optional) an optional name for the PricingModel
    notes (optional) an optional notes field for the PricingModel

  • FixedPrice — if the price of a Service is constant, no matter the duration a customer can reserve for, specify the FixedPrice type when creating a PricingModel for the Service. An example of such a Service would be a “Man with a Van” that charges for the entire day of work no matter whether the customer requested 2 hours or 8 hours of time on any given day.

    price price that will be charged for the entire Service
    currency (optional) currency of price. Defaults to USD
    name (optional) an optional name for the PricingModel
    notes (optional) an optional notes field for the PricingModel

Schedules

GoneBusy Schedules are no different from other types of schedules you may have encountered before: they start on a date, may end on a date, or possibly repeat indefinitely; falling between certain hours on particular days of the week. By defining a Schedule for a Service that is offered by a Resource, it is possible to advertise the available times the Service may be booked.

A Schedule is made up of a single or many TimeWindows, where the most important attributes of a TimeWindow are the following:

start_date what date it begins on
end_date what date it ends on (if any). no end_date means this scheduled window of time is valid forever
days the days of the week it falls on, such as Sunday through Saturday, everyday, weekends, etc.
start_time the earliest appointment time for the service
end_time when the last appointment must be finished by

There are additional, advanced features of a TimeWindow explained below.

You can find all supported operations on the Schedules API page.

Types of Recurring TimeWindows

In many cases, a period of advanced time will continue be offered in a recurring fashion. Many increments of recurrence for a TimeWindow are supported. Here is a listing of the supported types.

daily specify this kind of recurrence if a Service is offered every single day at the same time. this makes specifying days unnecessary
weekly something that recurs every week, such as a class
monthly something that recurs every month. its possible to specify that something recurs only the first week of each month as well
yearly this is useful for Services that happen on some date or stretch of days every year

Frequency

Frequency is a kind of multiplier for how the TimeWindow recurs. For example, a monthly TimeWindow with days = ‘weekend’ and frequency = ‘every_other’ and interval = ‘every’ implies that we are interested in a TimeWindow that actually recurs “every weekend every other month”.

Occurrence

Interval tells us exactly which instance of a TimeWindow’s recurrence we are interested in. For example, a monthly TimeWindow with days = ‘weekend’ and frequency = ‘every_other’ and interval = ‘2nd_to_last’ implies that we are interested in a TimeWindow that actually recurs “every 2nd to last weekend every other month”.

Search Results

GoneBusy offers a fully featured Search API for retrieving Services that meet search criteria such as Category name, geographic distance, rating, price, etc.

You can find all supported operations on the Search API page.

Users

Users are the people that participate on GoneBusy. If you haven't done so already, take a look at our explanation of how Users interact with GoneBusy on the Overview page.

The API allows a User to manage themselves or an Account Manager to operate on managed users in order to offer services, schedule availability, and various other operations.

You can find all supported operations on the Users API page.
